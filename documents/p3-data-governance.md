# P3 数据治理方案（轻量化数据治理）

## 1. 任务目标

将系统从"重平台化运营"降级为"轻阅读分发端"，保留必要数据能力，清理过期行为日志，冻结复杂画像运营能力。

---

## 2. 数据分类与保留策略

### 2.1 系统日志（保留）

| 数据表 | 保留期 | 用途 | 清理策略 |
|--------|--------|------|----------|
| `t_access_log` | 90 天 | 安全排障与基础访问统计 | 每日凌晨 3:00 自动清理 |
| `t_error_log` | 180 天 | 故障排查 | 每日凌晨 3:00 自动清理 |
| `t_search_history` | 90 天 | 搜索质量优化 | 每日凌晨 3:00 自动清理 |

### 2.2 小程序用户数据（保留 - 轻阅读能力）

| 数据表 | 保留期 | 用途 | 清理策略 |
|--------|--------|------|----------|
| `t_mp_browse_history` | 180 天 | 读者阅读记录 | 每日凌晨 3:00 自动清理 |
| `t_mp_reco_exposure` | 90 天 | 轻量推荐效果核对 | 每日凌晨 3:00 自动清理 |
| `t_mp_user_favorite` | 永久 | 用户收藏 | 用户删除时清理 |
| `t_mp_user_subscribe` | 永久 | 订阅状态 | 用户删除时清理 |
| `t_mp_user` | 永久 | 用户基础信息 | 用户删除时清理 |

### 2.3 冻结表（L2 已冻结 - 定期清理）

| 数据表 | 保留期 | 用途 | 清理策略 |
|--------|--------|------|----------|
| `t_mp_user_behavior` | 90 天 | 历史行为数据 | 每日凌晨 3:00 自动清理 |
| `t_mp_user_interest` | 90 天 | 历史兴趣数据 | 每日凌晨 3:00 自动清理 |
| `t_mp_user_tag` | 90 天 | 历史标签数据 | 每日凌晨 3:00 自动清理 |
| `t_mp_user_profile` | 90 天 | 历史画像数据 | 每日凌晨 3:00 自动清理 |

**说明**：这些表在 L2 阶段已冻结，不再新增写入。定期清理历史数据以释放存储空间，同时避免数据无限增长。

### 2.4 内容统计数据（保留）

| 数据表 | 保留期 | 用途 | 清理策略 |
|--------|--------|------|----------|
| `t_statistics_article_pv` | 365 天 | 内容表现回看 | 每日凌晨 3:00 自动清理 |

## 3. 代码实现

### 3.1 保留策略配置

文件：`sanmoo-server-go/internal/application/maintenance/service.go`

```go
var retentionPolicies = map[string]int{
    // 系统日志
    "t_access_log":     90,  // 访问日志保留 90 天，用于安全排障
    "t_error_log":      180, // 错误日志保留 180 天，用于故障排查
    "t_search_history": 90,  // 搜索历史保留 90 天，用于搜索质量优化

    // 小程序用户数据（轻阅读能力）
    "t_mp_browse_history": 180, // 阅读历史保留 180 天，读者阅读记录
    "t_mp_reco_exposure":  90,  // 推荐曝光记录保留 90 天，轻量推荐效果核对

    // 冻结表（L2 已冻结，不再新增写入，定期清理历史数据）
    "t_mp_user_behavior":  90, // 用户行为日志保留 90 天
    "t_mp_user_interest":  90, // 用户兴趣维度保留 90 天
    "t_mp_user_tag":       90, // 用户标签保留 90 天
    "t_mp_user_profile":   90, // 用户画像保留 90 天

    // 内容统计数据
    "t_statistics_article_pv": 365, // 文章日 PV 汇总保留 365 天，内容表现回看
}
```

### 3.2 时间字段映射

```go
timeField := "create_time"
switch tableName {
case "t_search_history":
    timeField = "search_time"
case "t_mp_reco_exposure":
    timeField = "exposed_at"
case "t_statistics_article_pv":
    timeField = "pv_date"
case "t_mp_user_behavior":
    timeField = "event_time"
case "t_mp_user_interest", "t_mp_user_tag", "t_mp_user_profile":
    timeField = "update_time"
}
```

### 3.3 清理机制

**自动清理**：
- 定时任务：每日凌晨 3:00 自动执行
- 执行频率：每分钟检查一次，到达指定时间执行清理
- 超时时间：10 分钟（防止清理任务占用过多资源）

**手动触发**：
```bash
curl -X POST http://localhost:28080/admin/maintenance/cleanup
```

## 4. 数据最小化原则

### 4.1 保留的数据

1. **用户核心功能数据**：收藏、浏览历史、订阅状态
2. **内容表现数据**：文章 PV 统计（用于内容运营回看）
3. **系统运维数据**：访问日志、错误日志（用于故障排查）

### 4.2 清理的数据

1. **过期行为日志**：超过 90 天的用户行为记录
2. **冻结表历史数据**：用户画像、兴趣、标签等已冻结能力的数据
3. **过期推荐曝光记录**：超过 90 天的推荐曝光日志

### 4.3 不再扩张的数据

根据 L2 阶段冻结策略，以下能力已冻结，不再新增写入：
- 复杂用户画像生成
- 自动行为标签打标
- 雷达图数据聚合
- 重运营型个性化推荐

---

## 5. 用户数据删除

小程序用户可通过 `DELETE /mp/user` 接口删除账号及所有相关数据：

- 用户基础信息（`t_mp_user`）
- 用户收藏（`t_mp_user_favorite`）
- 浏览历史（`t_mp_browse_history`）
- 订阅状态（`t_mp_user_subscribe`）
- 行为日志（`t_mp_user_behavior`）
- 兴趣画像（`t_mp_user_interest`）
- 用户标签（`t_mp_user_tag`）
- 用户六边形画像（`t_mp_user_profile`）

## 7. 风险与后续建议

### 7.1 风险

- 清理任务可能在数据量较大时执行较慢（已设置 10 分钟超时）
- 冻结表数据清理后无法恢复（建议定期备份）

### 7.2 后续建议

1. **监控清理效果**：定期检查清理报告，确认数据量在合理范围
2. **备份策略**：对重要数据进行定期备份
3. **性能优化**：如数据量持续增长，可考虑按月分表或归档
